# 「エージェント基盤を作れる人」とは何か

`LINE_WORKS_MCP_SERVER_PLAN.md`を実行する前に理解しておく、基礎概念の整理。

LLM(ChatGPTやClaude)そのものを作る人ではなく、**LLMが実際に外の世界で安全かつ確実に「仕事」ができるようにする配管・土台を作る人**のこと。

---

## 構成要素(重要度が高い順)

### 1. ツール設計(Tool/Function定義)
LLMに「何ができるか」を教える部分。例えば「LINE WORKSにメッセージを送る」という機能を、LLMが読んで理解できる形(JSON Schema)で定義する。ここが下手だと、LLMは何ができるか誤解して間違った呼び出し方をする。良い設計とは「1つの関数は1つのことだけをする」「パラメータの意味が名前だけで分かる」ようなAPI設計力そのもの。

### 2. プロトコル/標準化(MCPはここ)
ツールをアプリごとに個別実装するのではなく、共通規格で公開する部分。LINE WORKS MCPサーバーはまさにこれ。

### 3. オーケストレーション(実行の制御)
LLMが「次に何をすべきか」を決めた後、実際にどう実行するかの制御。単純な1つのループ(呼ぶ→結果を見る→また呼ぶ)なのか、複数のエージェントに仕事を分担させる(サブエージェント)のか。

### 4. 状態管理(メモリ・コンテキスト)
会話が長くなった時に何を覚えておくか、過去の情報をどう検索して持ってくるか(RAG)。

### 5. 実行環境の安全性(サンドボックス・権限)
LLMに「何でもできる」ようにすると危険なので、権限を絞る・実行環境を隔離する部分。「pushする前に確認を取る」「シークレットが見つかったら自動設定をスキップする」といった制御がこの層。

### 6. 観測性(ログ・トレース)
エージェントが「なぜその判断をしたか」を後から追えるようにする部分。

### 7. 評価・ガードレール
エージェントの出力が正しいか、危険でないかを継続的にチェックする仕組み。

---

## 生きた教材としての具体例

Claude Codeとのやり取り自体が、この7層の実例になっている。

| 層 | 実例 |
|---|---|
| 1. ツール設計 | `Bash`や`gh`コマンドの呼び出し |
| 2. MCP | `mcp__ccd_session__*`のような接頭辞のツール |
| 3. オーケストレーション | ワークフローを起動して裏で待つ、サブエージェントに調査を任せる |
| 4. 状態管理 | `memory`フォルダに会社情報を保存 |
| 5. 安全性 | 「pushする前に確認」「シークレット検出で自動停止」 |
| 6. 観測性 | ワークフローのログを見て何が起きたか確認 |

LINE WORKS MCPサーバーを作るということは、この仕組みの「LINE WORKS版の小さな相当品」を自分の手で作るということ。

---

## 学ぶ順番

1. MCPの公式ドキュメント(modelcontextprotocol.io)でプロトコルの仕様を読む(まず「2」を理解する)
2. 既存のシンプルなMCPサーバー実装(公式リポジトリのsample-servers)を1つ読んで写経する
3. 自分のLINE WORKS用ツール(4個)を実装する(ここで「1」の設計力が問われる)
4. Claude Desktopに繋いで実際に動かし、ログを見る習慣をつける(自然と「6」が身につく)

「作れる人」とは、7層すべてを1人で完璧にできる人という意味ではなく、**最低限1〜3を自分の手で組み立てた経験がある人**という意味。
