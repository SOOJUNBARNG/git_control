# LINE WORKS MCPサーバー 開発計画

「エージェント基盤を作れる人」として副業的に認知されるための、最初の具体的プロジェクト。`CAREER_ROADMAP.md`のAルート(AIエージェント副業)の第一弾。

対象範囲を「LINE WORKS」だけに絞り、`jinjer`や`Firebase`は後回しにする。

---

## なぜMCPサーバーなのか(仕組みの整理)

エージェント(Claude等のLLM)は単体では外の世界に触れられない。実際に動くのは以下の3層構造:

1. **モデル(LLM)**: 文脈から「次に何をすべきか」を判断するだけの存在
2. **ツール定義(スキーマ)**: JSON Schemaの形で「呼び出せる関数」がLLMに渡される
3. **ホスト/クライアント**: 実際に関数を実行し、結果をテキストでLLMに返す。これの繰り返しがエージェントの正体

MCP(Model Context Protocol)は②③を標準化するプロトコル。「LINE WORKS用のツール一式」をJSON-RPCベースの共通プロトコルで公開するプロセスを1つ作れば、Claude Code、Claude Desktop、他のエージェントフレームワークなど、どのMCPクライアントからも同じツールを再利用できる。

今の`LINE_LINEWORKS_GAS_SERVER_AI_CHATSERVICE`は、これを自前でベタ書きしている状態。それをMCPサーバーとして切り出すのは、「1つのアプリに埋め込んだロジック」を「誰でも繋げる汎用部品」に変える作業。

---

## スコープ(ツールは4個まで)

多機能にしない。最初はこれだけ:

- `send_message` — 指定チャンネル/ユーザーへのメッセージ送信
- `list_users` — 組織内ユーザー一覧取得
- `create_task` — タスク作成
- `get_talk_summary` — トーク履歴の要約取得(既存のLINE_WORKS_SUMMARYのロジックを流用)

実データ・実認証情報は一切含めない。モック/サンプルデータのみでデモを作る。ライセンスはMIT。

## 8週間プラン

### Week 1-2: 作る
- 公式のMCP TypeScript/Python SDKを使用
- 上記4ツールのみ実装
- 実認証情報は環境変数化し、リポジトリには含めない

### Week 3-4: 「作品」として仕上げる
- 英語READMEを書く(セットアップ手順、アーキテクチャ図、なぜ作ったか)
- 30秒程度のデモGIF/動画を1本作る(Claude Desktopに繋いで実際に動く様子)
- リポジトリ名・説明文を「業務ツール」ではなく「OSSプロダクト」らしく整える

### Week 5: 公式の場に載せる
- MCPサーバーの公式コミュニティリスト(`modelcontextprotocol/servers`、`awesome-mcp-servers`等)にPRを送り掲載してもらう

### Week 6: 発信する(1本だけ)
- Zenn(日本語)かdev.to(英語)のどちらかに、「なぜLINE WORKSのMCPサーバーを作ったか」を技術記事にする
- 量産しない。1本を丁寧に書く方が効く

### Week 7-8: 反応を見て次を決める
- GitHub Star数・記事の反応を見て、需要がある方向にだけ次の1個(jinjer or Firebase)を追加するか判断
- スカウトや副業の問い合わせが来始めたら、それが「認知された」の実質的なサイン
